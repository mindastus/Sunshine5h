## a) Online. Tee uusi varasto GitHubiin (tai Gitlabiin tai mihin vain vastaavaan palveluun). Varaston nimessä ja lyhyessä kuvauksessa tulee olla sana "sunshine". Aiemmin tehty varasto ei kelpaa. (Muista tehdä varastoon tiedostoja luomisvaiheessa, esim README.md ja GNU General Public License 3) Edistyneemmille voi olla hauskaa etsiä ja kokeilla jokin muu palvelu kuin Github.

## b) Dolly. Kloonaa edellisessä kohdassa tehty uusi varasto itsellesi, tee muutoksia omalla koneella, puske ne palvelimelle, ja näytä, että ne ilmestyvät weppiliittymään.

- Kloonasin aiemmin luomani sunshine‑varaston omalle koneelleni komennolla git clone. Tämän jälkeen tein paikallisesti muutoksia (esimerkiksi muokkasin README.md‑tiedostoa) ja tallensin ne Gitin normaalilla työflow’lla: git add, git commit ja git push. Kun päivitys oli työnnetty GitHubiin, tarkistin selaimessa, että muutokset näkyivät varaston weppiliittymässä, mikä vahvisti onnistuneen pushin.

## c) Doh! Tee tyhmä muutos gittiin, älä tee commit:tia. Tuhoa huonot muutokset ‘git reset --hard’. Huomaa, että tässä toiminnossa ei ole peruutusnappia.

- Tein tarkoituksella huonon muutoksen paikalliseen Git‑varastoon, mutta en tehnyt commitia, jotta muutos jäi pelkästään working treeen. Kun halusin perua koko sotkun, käytin komentoa git reset --hard, joka palautti projektin viimeisimmän commitin tilaan ja poisti kaikki tekemäni tallentamattomat muutokset. Tämä komento on lopullinen eikä sille ole peruutusnappia, joten sitä kannattaa käyttää vain silloin, kun varmasti haluaa hävittää kaikki paikalliset muutokset.

## d) Tukki. Tarkastele ja selitä varastosi lokia. Näytä myös, mitä muutoksia tiedostoihin on tehty. Tarkista, että nimesi ja sähköpostiosoitteesi näkyy haluamallasi tavalla ja korjaa tarvittaessa.

- Tarkastelin varastoni commit‑historiaa komennolla: git log.Lokista näin jokaisen commitin tunnisteen (SHA‑1), tekijän nimen ja sähköpostiosoitteen sekä commit‑viestin. Varmistin samalla, että Git käyttää oikeaa nimeä ja sähköpostia, ja korjasin asetukset tarvittaessa komennoilla:

git config --global user.name "Oma Nimi"
git config --global user.email "oma@example.com"

Seuraavaksi tarkastelin, mitä muutoksia commitit sisältävät komennolla:
git log -p

Tämä näytti rivikohtaiset erot (diffit), eli mitä rivejä tiedostoihin oli lisätty tai poistettu. Näin pystyin selkeästi osoittamaan, miten varasto oli kehittynyt commitista toiseen.

## e) Gitanbile. Laita Ansible-kansio versionhallintaan. Tee jokin muutos, aja ansiblella, tallenna versio (commit).

- Lisäsin oman Ansible‑kansioni versionhallintaan siirtymällä sen juureen ja alustamalla Gitin komennolla git init. Tämän jälkeen lisäsin kaikki kansiossa olevat tiedostot versionhallintaan komennolla git add . ja tein ensimmäisen commitin. Muokkasin yhtä Ansible‑tiedostoa (esimerkiksi site.yml tai roolin tasks/main.yml), ajoin muutoksen Ansiblella varmistaakseni, että se toimii, ja tallensin muutoksen uudella commitilla. Lopuksi työnsin commitit GitHubiin git push‑komennolla, jolloin Ansible‑kansion päivitykset ilmestyivät varastoon.
